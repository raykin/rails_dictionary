# rails_dictionary Roadmap

**Core principle: intuitive API. Use naturally, learn nothing.**

Breaking release. Not backward compatible with v0.2.

## Target use case

Applications where dictionary data is **shared across multiple models** and models reference **many lookup fields (10+)**. For example, enrollment forms, immigration applications, or survey systems where cities, degree types, visa types etc. are referenced by Student, University, Company, and other models.

If only one model needs a lookup field and nothing else references it, a plain string column is sufficient — no table or gem needed. The gem's value scales with: **many lookup fields × shared across multiple models**.

---

## v1.0

### 1. One table, simple schema

Remove `dict_types` table and `DictType` model. Single table:

```
dictionaries
- id
- name
- category (e.g. "city", "degree", "visa_type")
```

### 2. Clean API

Remove `method_missing`, Array monkey-patch, locale feature. Replace with `define_method`.

```ruby
# Dictionary model
Dictionary.city                      # cached by default
Dictionary.city(query: true)         # hits DB
Dictionary.categories                # list all categories

# CRUD
Dictionary.city.add("New York")
Dictionary.city.remove("New York")

# Consumer model
class Student < ApplicationRecord
  uses_dictionary :city, :degree, :major
end
```

### 3. Auto caching

Cache primitive data (ids, names) by default. No manual cache management.

### 4. Form-ready output

`Dictionary.city` returns data in a format ready for `options_for_select`.

### 5. GitHub CI

Set up GitHub Actions for automated testing.

## v1.1

### 6. Generator

### 7. `Dictionary.categories`
